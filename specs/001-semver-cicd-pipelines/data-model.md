# Data Model: Semantic Versioning CI/CD Pipelines

**Feature**: 001-semver-cicd-pipelines
**Date**: 2026-03-26

This feature is a GitHub Actions reusable workflow. There is no traditional
application data model (no database, no persistent entities managed by application
code). The "data" is workflow configuration, Git refs, and GitHub API objects.
This document describes those entities and their relationships.

---

## Entity 1: Reusable Workflow (`semver-release.yml`)

The primary artifact. A GitHub Actions workflow file that exposes a `workflow_call`
trigger and encapsulates all release logic.

**Location**: `.github/workflows/semver-release.yml`

**Inputs** (declared under `on.workflow_call.inputs`):

| Field | Type | Required | Default | Description |
|-------|------|----------|---------|-------------|
| `release-branch` | string | no | `main` | The branch that triggers a release when pushed to |
| `tag-prefix` | string | no | `v` | Prefix prepended to version numbers (e.g., `v` → `v1.2.3`) |

**Outputs** (declared under `on.workflow_call.outputs`):

| Field | Type | Description |
|-------|------|-------------|
| `release-created` | string (`"true"`/`"false"`) | Whether a new release was published this run |
| `tag-name` | string | The full version tag created (e.g., `v1.2.3`); empty if no release |
| `major-tag` | string | The major pointer tag updated (e.g., `v1`); empty if no release |

**Permissions** (declared at job level):

| Permission | Scope |
|------------|-------|
| `contents` | `write` |

**Release engine**: `cycjimmy/semantic-release-action` @ `b12c8f6015dc215fe37bc154d4ad456dd3833c90` (v6.0.0)

## Entity 2: Version Tag

A Git tag created by the pipeline on the `main` branch commit that triggered the release.

**Format**: `{tag-prefix}{MAJOR}.{MINOR}.{PATCH}` — e.g., `v1.2.3`

**Lifecycle**:
1. Created by `cycjimmy/semantic-release-action` when it detects releasable commits on push to `main`.
2. Points to the HEAD commit of the push that triggered the release.
3. Immutable once created — never force-updated.

**Initial state**: If no prior version tags exist, the first release creates `v1.0.0`.

**Validation rules**:
- MAJOR increment: at least one `feat!:` commit or `BREAKING CHANGE:` footer since last tag.
- MINOR increment: at least one `feat:` commit since last tag (and no breaking changes).
- PATCH increment: at least one `fix:` or `perf:` commit since last tag (and no feat or breaking changes).
- No release: only `chore:`, `docs:`, `style:`, `test:`, `build:`, `ci:` commits since last tag.

---

## Entity 3: Major Pointer Tag

A mutable Git tag that always points to the latest release on a given major version
line. Enables consumers to pin `@v1` and receive all patch/minor updates automatically.

**Format**: `{tag-prefix}{MAJOR}` — e.g., `v1`

**Lifecycle**:
1. Created on first release of a new major version (e.g., `v1.0.0` creates `v1`).
2. Force-updated on every subsequent MINOR or PATCH release within that major line.
3. A new MAJOR release (e.g., `v2.0.0`) creates a new pointer (`v2`) and leaves the
   previous pointer (`v1`) frozen at the last `v1.x.y` release.

**Relationships**: One Major Pointer Tag per major version line. Many Version Tags
map to one Major Pointer Tag.

---

## Entity 4: GitHub Release

A GitHub Release record attached to a Version Tag. Created by
`cycjimmy/semantic-release-action` in the same run that creates the tag.

**Fields**:

| Field | Value |
|-------|-------|
| `tag_name` | The version tag (e.g., `v1.2.3`) |
| `name` | Auto-generated by semantic-release (e.g., `v1.2.3`) |
| `body` | Auto-generated changelog from conventional commit messages |
| `draft` | `false` |
| `prerelease` | `false` |

**Relationships**: One GitHub Release per Version Tag.

## Entity 5: Consumer Release Config (`.releaserc.json`) — Optional

An optional JSON file in a consuming repository that overrides the inline plugin
configuration provided by the reusable workflow.

**Location in consumer repo**: `.releaserc.json` (root; semantic-release discovers
this by convention automatically — no explicit path input required).

**When to provide**: Only needed for advanced configuration (custom changelog
sections, non-standard bump strategies, monorepos). Standard single-package repos
do not need this file — the reusable workflow's inline `extra_plugins` config handles
the standard case.

**Example override content**:
```json
{
  "branches": ["main"],
  "plugins": [
    ["@semantic-release/commit-analyzer", { "preset": "conventionalcommits" }],
    ["@semantic-release/release-notes-generator", { "preset": "conventionalcommits" }],
    ["@semantic-release/github"]
  ]
}
```

**Relationships**: Zero or one config file per consumer repo. If present at repo root,
semantic-release automatically uses it instead of the inline `with:` block configuration.

## Entity 6: Consumer Workflow File

A GitHub Actions workflow file in a consuming repository that calls the reusable
workflow via `uses:`.

**Minimum viable consumer workflow** (< 20 lines, satisfying SC-001):
```yaml
name: Release

on:
  push:
    branches: [main]

jobs:
  release:
    permissions:
      contents: write
    uses: jmckenzie17/homeschoolio-shared-actions/.github/workflows/semver-release.yml@v1
    secrets: inherit
```

Note: `permissions: contents: write` must be declared on the calling job — the
reusable workflow cannot self-elevate its `GITHUB_TOKEN` scope.

**Relationships**: Lives in the consumer repo. References the Reusable Workflow
(Entity 1) by path and tag.

---

## Entity 7: Self-Release Caller Workflow (`release.yml`)

The workflow in this repo that versions `homeschoolio-shared-actions` itself using
the reusable workflow via local path reference.

**Location**: `.github/workflows/release.yml`

**Trigger**: `on: push: branches: [main]`

**Concurrency**: `group: release`, `cancel-in-progress: false`

**Key properties**:
- References the reusable workflow via local path: `uses: ./.github/workflows/semver-release.yml`
- No bootstrap dependency — local path resolves from the same commit.
- Declares `permissions: contents: write` on the calling job.
